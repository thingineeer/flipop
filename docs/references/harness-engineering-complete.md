## 1. 본문 요약 (섹션별)

### 1.1 도입부: 실험의 배경

OpenAI 팀은 5개월간 수동으로 코드를 한 줄도 작성하지 않고 소프트웨어 제품의 내부 베타를 구축하고 출시하는 실험을 진행했다. 모든 코드(애플리케이션 로직, 테스트, CI 구성, 문서화, 관측 가능성, 내부 툴링)가 Codex에 의해 작성되었으며, 수작업 대비 약 1/10의 시간이 소요된 것으로 추정한다.

### 1.2 사람이 조정하고, 에이전트가 수행

엔지니어링 팀의 주된 업무가 코드 작성에서 **환경 설계, 의도 명시, 피드백 루프 구축**으로 전환되었다. 약 백만 라인의 코드를 출시하는 데 몇 주가 소요되었고, 3명의 엔지니어가 약 1,500개의 PR을 열고 병합했다 (1인당 하루 평균 3.5개 PR). 팀이 7명으로 성장하면서 처리량도 증가했다.

핵심 철학: **수동으로 작성한 코드가 없다**.

### 1.3 빈 git 리포지터리로 시작

- 첫 번째 커밋: 2025년 8월 말
- 초기 스캐폴드(리포지터리 구조, CI 구성, 서식 규칙, 패키지 관리자, 앱 프레임워크)는 GPT-5를 사용한 Codex CLI에서 생성
- 초기 AGENTS.md 파일도 Codex가 직접 작성
- 처음부터 리포지터리는 에이전트에 의해 형성됨

### 1.4 엔지니어의 역할 재정의

사람이 직접 코딩하지 않으므로 **시스템, 스캐폴딩, 레버리지** 중심의 엔지니어링 작업이 필요했다.

- 초기 진행이 더뎠던 이유: Codex의 역량 부족이 아니라 **환경이 미비**했기 때문
- 큰 목표를 작은 빌딩 블록으로 세분화 -> 에이전트가 이를 조합하도록 유도 -> 더 복잡한 작업 해결
- 엔지니어는 거의 전적으로 **프롬프트**를 통해 시스템과 상호작용
- **Ralph Wiggum Loop** 패턴: Codex가 자체 변경사항을 검토하고, 에이전트 검토를 요청하고, 피드백에 응답하고, 만족할 때까지 반복
- PR 검토도 점차 에이전트 간 처리로 전환

### 1.5 애플리케이션의 가독성 향상

코드 처리량 증가로 인적 QA에 병목 발생. 에이전트에 더 많은 기능을 부여하기 위해:

- **git worktree별 앱 부팅**: 변경사항마다 독립 인스턴스 실행
- **Chrome DevTools Protocol 연결**: DOM 스냅샷, 스크린샷, 탐색 작업용 스킬 구축 -> 버그 재현, 수정 검증, UI 동작 추론 가능
- **로컬 관측 가능성 스택**: LogQL로 로그 쿼리, PromQL로 메트릭 쿼리 -> 각 워크트리별 격리
- 한 번의 Codex 실행으로 6시간 이상(사람이 잠자는 동안) 작업 수행

### 1.6 리포지터리 지식을 기록 시스템으로 만듦

컨텍스트 관리가 가장 큰 과제. "하나의 큰 AGENTS.md" 접근 방식은 실패:

- 컨텍스트는 희소 리소스 -> 거대한 지침으로 핵심 제약 조건 놓침
- 지침이 너무 많으면 지침이 되지 않음
- 순식간에 낡은 규칙의 무덤으로 변함
- 기계적 점검이 어려움

**해결책: AGENTS.md를 목차로 취급**

```
AGENTS.md          (약 100줄, 맵 역할)
ARCHITECTURE.md
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   ├── new-user-onboarding.md
│   └── ...
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
│   ├── uv-llms.txt
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

- 설계 문서: 검증 상태와 에이전트 우선 운영 원칙, 핵심 신념 포함
- 아키텍처 문서: 도메인과 패키지 레이어링의 최상위 맵
- 품질 문서: 각 도메인/레이어에 등급 매기고 격차 추적
- 실행 계획: 일급 아티팩트로 취급, 진행 상황과 의사결정 로그 포함
- **기계적 시행**: 린터와 CI 작업으로 지식 베이스 최신 상태 검증
- **doc-gardening 에이전트**: 오래된 문서를 검토하여 수정 PR 자동 생성

### 1.7 에이전트의 가독성이 목표

- 리포지터리가 전적으로 에이전트에 의해 생성 -> Codex의 가독성에 최적화
- 에이전트 관점에서 컨텍스트 내 액세스할 수 없는 것은 존재하지 않는 것
- Google Docs, 채팅, 머릿속 지식은 접근 불가 -> 버전 관리되는 아티팩트만 접근 가능
- Slack 토론에서 합의된 내용도 리포지터리에 기록해야 함
- **"지루한" 기술 선호**: 결합성, API 안정성, 학습 데이터 내 표현이 좋아 에이전트가 모델링하기 쉬움
- 불투명한 외부 라이브러리보다 자체 구현이 더 저렴한 경우도 있음 (예: p-limit 대신 자체 map-with-concurrency 헬퍼)

### 1.8 아키텍처 및 취향 강제 적용

- 문서화만으로는 일관성 유지 불가 -> **불변 조건을 기계적으로 강제**
- 엄격한 아키텍처 모델: 각 비즈니스 도메인은 고정된 계층으로 분리
  - `Types -> Config -> Repo -> Service -> Runtime -> UI`
  - 교차 문제(인증, 커넥터, 텔레메트리, 기능 플래그)는 Providers를 통해 유입
  - 그 외 모든 것은 허용되지 않으며 기계적으로 강제
- **맞춤형 린터와 구조적 테스트**로 규칙 적용:
  - 구조화된 로깅
  - 스키마 및 유형의 명명 규칙
  - 파일 크기 제한
  - 플랫폼별 안정성 요구 사항
- 린트 오류 메시지에 **에이전트용 수정 지침을 주입**
- 경계에서 데이터 형태를 파싱 (Parse, don't validate 원칙)
- 중앙에서 경계 설정, 로컬에서 자율성 허용
- 인간 취향은 리뷰 코멘트, 리팩터링 PR, 사용자 버그를 통해 지속적으로 피드백

### 1.9 병합 철학을 변화시키는 처리량

- 최소한의 차단 병합 게이트로 운영
- PR은 수명이 짧음
- 테스트 불안정성은 후속 실행으로 해결
- 에이전트 처리량이 사람의 주의력을 초과하는 시스템에서는 **수정 비용이 저렴하고 대기 시간이 비쌈**

### 1.10 "에이전트 생성"의 실제 의미

에이전트가 생성하는 것:
- 제품 코드 및 테스트
- CI 구성 및 릴리스 툴링
- 내부 개발자 도구
- 문서화 및 설계 내역
- 평가 하네스
- 리뷰 코멘트 및 응답
- 리포지터리 자체를 관리하는 스크립트
- 생산 대시보드 정의 파일

사람의 역할: 업무 우선순위 설정, 사용자 피드백을 수용 기준으로 변환, 결과 검증.
에이전트가 어려움을 겪으면 도구/가드레일/문서화 누락을 파악 -> Codex에서 수정사항 작성.

### 1.11 자율성 수준의 증가

리포지터리가 임계점을 넘어 에이전트가 새 기능을 처음부터 끝까지 주도 가능:

한 번의 프롬프트로:
1. 코드베이스 현재 상태 검증
2. 보고된 버그 재현
3. 실패 상황 동영상 녹화
4. 수정사항 구현
5. 앱 실행하여 수정 검증
6. 해결책 동영상 녹화
7. PR 열기
8. 에이전트/사람 피드백에 응답
9. 빌드 실패 감지 및 수정
10. 판단 필요 시 사람에게 에스컬레이션
11. 변경사항 병합

### 1.12 엔트로피 및 가비지 컬렉션

- 에이전트는 기존 패턴(불균일하거나 최적 상태가 아닌 것 포함)을 복제 -> 시간이 지나면 드리프트 발생
- 초기에 수동 처리 (매주 금요일 "AI 슬로프" 정리) -> 확장 불가
- **해결책: "황금 원칙"을 리포지터리에 인코딩**
  - 직접 만든 헬퍼보다 공유 유틸리티 패키지 선호
  - "YOLO-style" 데이터 탐색 금지 -> 경계 검증 또는 유형 지정 SDK 사용
- **Codex 백그라운드 작업**: 정기적으로 편차 검사, 품질 등급 업데이트, 리팩터링 PR 생성
- **가비지 컬렉션 비유**: 기술 부채를 조금씩 꾸준히 갚는 것이 한꺼번에 갚는 것보다 효과적

### 1.13 아직 배우고 있는 점

- 완전한 에이전트 생성 시스템에서 아키텍처 일관성이 수년에 걸쳐 어떻게 진화하는지 미지수
- 사람의 판단이 가장 큰 영향력을 발휘하는 지점과 그 인코딩 방법을 학습 중
- 모델 기능 향상에 따른 시스템 발전 방향도 불확실
- **분명해진 것**: 규율은 코드보다 **스캐폴딩**에서 더 중요

## 2. 하이퍼링크 리소스 분석

### 2.1 Ralph Wiggum Loop
- **URL**: https://ghuntley.com/loop/
- **저자**: Geoffrey Huntley

소프트웨어 개발을 전통적인 수직적 구축에서 반복적이고 순환적인 접근 방식으로 전환하는 개념이다. 소프트웨어를 "도자기 물레 위의 점토"처럼 취급하여, 문제가 생기면 처음부터 다시 시작하는 대신 루프로 돌려보내 개선한다.

핵심 원칙:
- **루프 마인드셋**: 소프트웨어를 정적 구조가 아닌 반복 개선 대상으로 봄
- **모놀리식 아키텍처**: 단일 프로세스, 단일 리포지터리에서 루프당 하나의 작업 수행
- **인간-컴퓨터 파트너십**: 엔지니어는 루프 자체를 프로그래밍하고 실행을 감독
- **운영 프로세스**: 사양/목표 설정 -> 컨텍스트 엔지니어링으로 루프 실행 -> 실패 모니터링 -> 필요시 수동 개입 -> 재발 방지 솔루션 구축

### 2.2 AGENTS.md
- **URL**: https://agents.md/

AI 코딩 에이전트를 위한 표준화된 오픈 포맷 마크다운 파일이다. "에이전트를 위한 README"로, 에이전트 전용 지침을 예측 가능한 위치에 제공한다.

핵심 정보:
- **채택률**: 60,000개 이상의 오픈소스 프로젝트에서 사용 중
- **지원 도구**: OpenAI Codex, Google Jules, Aider, VS Code, GitHub Copilot, Cursor 등
- **권장 섹션**: 프로젝트 개요, 빌드/테스트 명령, 코드 스타일, 테스트 지침, 보안 고려사항, 커밋 메시지 형식
- **모노레포 지원**: 하위 디렉터리에 별도 AGENTS.md 배치, 가장 가까운 파일이 우선
- **동적 문서**: 프로젝트 발전에 따라 업데이트 필요

### 2.3 ARCHITECTURE.md
- **URL**: https://matklad.github.io/2021/02/06/ARCHITECTURE.md.html
- **저자**: matklad (rust-analyzer 관리자)

10k-200k 라인 규모의 오픈소스 프로젝트에 README, CONTRIBUTING과 함께 추가해야 할 고수준 아키텍처 문서이다.

핵심 통찰:
- 기여자와 핵심 개발자의 가장 큰 차이는 프로젝트의 **물리적 아키텍처에 대한 지식**
- 패치 작성에 2배 시간이 걸리지만, **코드 변경 위치 파악에 10배** 시간이 걸림
- 핵심 개발자는 "멘탈 맵"을 가지고 있어 순차적으로 읽지 않고 바로 해당 위치로 점프

작성 방법:
1. 문제에 대한 조감도로 시작
2. 대략적인 코드맵 작성 (모듈과 관계)
3. "나라의 지도이지, 주(state)별 지도의 모음이 아님"
4. 중요 파일/모듈/타입의 이름을 지정하되 직접 링크는 피함
5. 아키텍처 불변 조건을 명시적으로 호출
6. 레이어 간 경계를 지적
7. 교차 관심사를 별도 섹션으로 추가
8. 짧게 유지하고 연 1-2회 재방문

### 2.4 Codex 실행 계획 (ExecPlans)
- **URL**: https://cookbook.openai.com/articles/codex_exec_plans (-> https://developers.openai.com/cookbook/articles/codex_exec_plans)

AI 코딩 에이전트가 복잡한 멀티-아워 문제 해결 작업을 수행하도록 안내하는 상세한 설계 문서이다.

비타협적 요구사항:
- **자체 완결성**: 리포지터리에 대한 사전 지식 없이 구현에 필요한 모든 지식을 포함
- **살아있는 문서**: 발견, 설계 결정, 진행 업데이트와 함께 진화
- **관찰 가능한 결과**: 작동하는 동작을 생산해야 함 (단순 코드 변경이 아님)
- **평이한 언어**: 모든 전문 용어를 평이한 언어로 정의
- **초보자 친화**: 완전한 초보자도 엔드투엔드 구현 성공 가능

필수 섹션:
- **Progress**: 타임스탬프 포함 체크박스 목록
- **Surprises & Discoveries**: 예상치 못한 동작, 성능 인사이트, 버그
- **Decision Log**: 근거와 날짜를 포함한 모든 중요 선택 기록
- **Outcomes & Retrospective**: 성과 요약, 남은 작업, 교훈

핵심 원칙:
- **오리엔테이션**: 전체 리포지터리 상대 경로, 정확한 함수명, 명시적 가정 포함
- **멱등성**: 단계가 반복 가능하고 드리프트/손상 없이 재실행 가능
- **검증**: 예상 출력을 포함한 포괄적 테스트 지침
- **앵커링**: 검증 가능한 동작으로 수용 기준 표현

### 2.5 Aardvark (에이전틱 보안 리서처)
- **URL**: https://openai.com/ko-KR/index/introducing-aardvark/
- **날짜**: 2025년 10월 30일

GPT-5가 구동하는 에이전틱 보안 리서처로, 개발자와 보안팀이 보안 취약점을 대규모로 파악하고 수정할 수 있도록 지원한다.

작동 방식 (다단계 파이프라인):
1. **분석**: 전체 저장소 분석 -> 위협 모델 생성
2. **커밋 스캔**: 새 코드 커밋 시 취약점 스캔, 기존 기록도 스캔
3. **검증**: 격리된 샌드박스에서 취약점 트리거하여 악용 가능성 확인
4. **패치**: Codex와 결합하여 취약점 수정 패치 생성

성과:
- 벤치마크에서 잘 알려진 취약점의 **92% 식별**
- 오픈소스 프로젝트에서 10개의 CVE 발견 및 책임 있는 공개
- OpenAI 약 1.2%의 커밋에서 버그 발생

Harness Engineering에서의 맥락: Codex와 함께 코드베이스에서 작업하는 또 다른 에이전트로, 에이전트가 검사하고 검증하며 직접 수정할 수 있는 형태로 시스템을 구축하면 다양한 에이전트에서 레버리지를 높일 수 있음.

### 2.6 Parse, Don't Validate
- **URL**: https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/
- **저자**: Alexis King

입력을 검증하고 정보를 버리는 대신, 더 정밀한 타입으로 파싱하여 잘못된 상태를 구조적으로 불가능하게 만드는 패턴이다.

핵심 아이디어:
- **검증 vs 파싱**: 검증은 조건을 확인하되 지식을 버림. 파싱은 조건을 확인하고 위반이 구조적으로 불가능한 형태로 데이터를 변환
- **타입 시스템을 증명으로 활용**: `NonEmpty a` 같은 타입은 불변 조건의 증명
- **"Shotgun parsing" 방지**: 파싱과 처리 코드를 혼합하는 안티패턴을 피하고, 시스템 경계에서 파싱을 완료

실용 전략:
1. 원하는 데이터에 맞는 함수 작성
2. `m ()` 반환 함수 의심 (검증을 숨기고 있을 수 있음)
3. 시스템 경계로 파싱 밀어내기
4. 완벽한 타입 인코딩이 어려우면 스마트 생성자와 추상 newtype 사용

Harness Engineering에서의 맥락: Codex가 경계에서 데이터 형태를 파싱하도록 원했으며 (Zod를 선호하지만 특정 라이브러리는 지정하지 않음), 이는 에이전트가 생성한 코드의 정확성을 높이는 핵심 원칙.

### 2.7 AI Is Forcing Us To Write Good Code
- **URL**: https://bits.logic.inc/p/ai-is-forcing-us-to-write-good-code

AI 에이전트가 이전에는 선택사항이었던 엄격한 코드 품질 관행을 필수로 만들고 있다는 주장이다.

5가지 필수 관행:
1. **100% 코드 커버리지**: 버그 방지 메트릭이 아닌, 에이전트가 모든 라인의 동작을 검증하는 수단. 100%에서 "위상 변화" 발생 - 미커버 라인 = 새로 추가된 코드
2. **사려 깊은 디렉터리 구조**: 파일 조직이 LLM의 인터페이스. `./billing/invoices/compute.ts` 같은 이름이 목적을 전달
3. **작고 잘 범위가 정해진 파일**: 에이전트가 큰 파일을 컨텍스트 로딩 시 잘라냄. 완전 로딩 가능한 크기 유지
4. **빠르고 임시적인 개발 환경**: 빠른 피드백 루프로 에이전트를 "짧은 목줄에 묶음". 10,000+ 테스트 어설션을 약 60초에 실행
5. **엔드투엔드 타입 시스템**: TypeScript, OpenAPI, PostgreSQL을 가로지르는 강한 타이핑. `UserId`, `SignedWebhookPayload` 같은 의미 있는 이름

Harness Engineering에서의 맥락: 엄격한 경계와 예측 가능한 구조를 갖춘 환경에서 에이전트가 가장 효과적이라는 주장을 뒷받침하는 외부 레퍼런스.

## 3. Flutter 모바일 게임 프로젝트에 적용 가능한 핵심 테이크어웨이

### 3.1 즉시 적용 가능

#### AGENTS.md + ARCHITECTURE.md 도입
- 프로젝트 루트에 짧은 `AGENTS.md` (100줄 이내) 작성: 빌드 명령, 테스트 명령, 디렉터리 맵, 코딩 규칙
- `ARCHITECTURE.md` 작성: Flutter 프로젝트의 레이어 구조 (presentation -> domain -> data), 주요 모듈, 의존성 방향 명시
- AI 에이전트(Claude Code, Codex, Cursor 등) 사용 시 맥락 제공이 크게 개선됨

#### 구조화된 docs/ 디렉터리
```
docs/
├── design-docs/        # 설계 결정 기록
├── product-specs/      # 제품 사양
├── exec-plans/         # 실행 계획 (active/completed)
│   └── tech-debt-tracker.md
└── references/         # 외부 레퍼런스
```

#### 엄격한 레이어 아키텍처 강제
- Flutter에 적합한 레이어: `Models -> Config -> Repository -> Service -> Provider -> UI`
- `analysis_options.yaml`과 커스텀 lint 규칙으로 의존성 방향 강제
- 비즈니스 도메인별 패키지 분리 (예: game_logic, ad_management, user_profile)

### 3.2 에이전트 친화적 코드베이스 구축

#### Parse, Don't Validate 원칙 적용
- Dart의 강한 타입 시스템 활용: `sealed class`, `enum`, `typedef`로 잘못된 상태를 구조적으로 불가능하게
- API 응답, Firebase 데이터 등 외부 데이터는 경계에서 즉시 타입 안전한 모델로 파싱
- `freezed` 패키지로 불변 데이터 클래스 + Union type 활용

#### 파일 크기와 구조 관리
- 파일당 200줄 이내 유지 (에이전트의 컨텍스트 윈도우 최적화)
- 의미 있는 디렉터리/파일 이름 사용: `lib/features/game/presentation/game_board_screen.dart`
- 하나의 파일에 하나의 책임

#### 테스트 커버리지 우선
- 게임 로직(점수 계산, 턴 관리 등)에 100% 단위 테스트 커버리지 목표
- Widget 테스트로 UI 동작 검증
- 테스트가 빠르게 실행되도록 유지 (에이전트 피드백 루프 단축)

### 3.3 중장기 적용

#### 기술 부채 가비지 컬렉션
- `docs/exec-plans/tech-debt-tracker.md`에 알려진 기술 부채 추적
- 정기적 리팩터링: 에이전트에 작은 범위의 정리 작업을 프롬프트
- 나쁜 패턴이 퍼지기 전에 매일 포착하고 해결

#### 커스텀 Lint 규칙
- Dart의 `custom_lint` 패키지 활용
- 오류 메시지에 수정 방법 포함 (에이전트가 자동으로 수정 가능하도록)
- 명명 규칙, 파일 구조, 금지된 패턴 등을 정적으로 강제

#### 관측 가능성 (Observability)
- Firebase Crashlytics + Analytics로 에이전트가 쿼리 가능한 메트릭 구축
- 구조화된 로깅으로 에이전트가 로그를 파싱하고 추론 가능하게

### 3.4 마인드셋 전환

1. **코드 작성자에서 환경 설계자로**: AI 에이전트가 효과적으로 작업할 수 있는 환경(가드레일, 문서, 테스트, 린트) 구축에 집중
2. **리포지터리가 곧 진실의 원천**: 모든 결정, 규칙, 컨텍스트를 리포지터리 안에 문서화. 머릿속이나 채팅에만 있으면 에이전트가 접근 불가
3. **수정 비용 < 대기 비용**: 빠르게 병합하고, 문제가 생기면 빠르게 수정. 완벽한 PR을 기다리기보다 빠른 반복이 중요
4. **에이전트가 어려움을 겪으면 환경의 문제**: 에이전트 실패는 도구/문서/가드레일의 부족을 알리는 신호
